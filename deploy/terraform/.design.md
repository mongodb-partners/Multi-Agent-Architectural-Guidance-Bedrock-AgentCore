# Terraform Infrastructure Design Decisions

Last updated: 2026-05-20.

This document captures the **why** behind non-obvious Terraform choices. For the
**what**, read the `.tf` files — they're the source of truth. For a flat
catalog of every module (inputs, outputs, consumers) see
[`docs/reference/terraform-modules.md`](../../docs/reference/terraform-modules.md);
for the cross-stack SSM contract see
[`docs/reference/ssm-parameters.md`](../../docs/reference/ssm-parameters.md).

---

## Four root configs: network + shared + local + ec2

**Decision:** The stack is split into four independent root configurations. `envs/network` + `envs/shared` are singletons; `envs/ec2` and `envs/local` are per-project / per-laptop.

| Env | Scope | Singleton scope | Deploy script | State key |
|---|---|---|---|---|
| `envs/network` | Shared VPC + 2x public + 2x private subnets + IGW + Atlas connectivity (Interface VPCE in `NETWORK_MODE=privatelink`, or VPC peering + Atlas Private DNS for Peering in `NETWORK_MODE=peering`) + Atlas-PL SG + `network_mode` SSM canary + SSM publish | account + region | `deploy-network.sh` | `<SHARED_VPC_NAME>/<region>/network/terraform.tfstate` |
| `envs/shared`  | Voyage SageMaker endpoint (when `EMBEDDINGS_PROVIDER=voyage`); CloudWatch log groups (`/<SHARED_RESOURCE_PREFIX>/<env>/{api,ui,mcp,agentcore,otel,otel-atlas}`); fleet+mongo+cost+atlas dashboards + alarms + metric filters + saved queries; Bedrock invocation logging (account-scoped singleton config) | account + region + environment | `deploy-shared.sh` | `<env>/shared/terraform.tfstate` |
| `envs/local`   | Cognito + Bedrock KB + Secrets Manager for laptop dev | per-laptop | `deploy-local.sh` | `<env>/terraform.tfstate` |
| `envs/ec2`     | Atlas M10 + Bedrock KB (+ KB PL/peering NLB when enabled) + EC2 + ECR + Cognito + AgentCore (Memory / Gateway / 4 chat Runtimes + 1 MongoDB MCP Runtime) + per-cluster Route 53 zone (PL mode only) + per-project ADOT sidecar | per-project (any number can coexist) | `deploy-project.sh` | `<env>/ec2/terraform.tfstate` |

All four share the **same S3 state bucket** (`${project_name}-${environment}-${account_id}`)
but use **different state keys**, so they can be applied and destroyed
independently. **Apply order on a clean account is `network` → `shared` → `ec2`** (or `local`)
because per-project envs read the shared VPC + Atlas + observability details
from SSM Parameter Store published by the upstream envs (see "Shared network
is consumed via SSM" below). The `deploy-full-with-privatelink.sh` and
`deploy-full-with-vpc-peering.sh` orchestrators probe SSM canaries for each
upstream stack and run the missing ones automatically.

**Why this split?** Originally `envs/ec2` provisioned its own VPC + Atlas
PrivateLink VPCE. That works for one project but means every additional project
in the same account/region pays for a fresh VPC and a fresh Atlas Interface
VPCE (~$7-10/AZ/month per VPCE). Atlas itself only allows **one endpoint
service per (Atlas project, region)**, so the Atlas-side resource was already
shared via the discover-or-create script — collapsing the AWS-side VPC to one
just made the topology match Atlas's constraint.

Per-project isolation is preserved through:
- **Per-project security groups** (project SG on EC2 + project SG on Lambda MCP).
- **Per-project IAM roles** (EC2, Lambda exec, AgentCore Gateway, Bedrock KB).
- **Per-project Atlas cluster + DB user**.
- **Per-project Route 53 private zone** for that cluster's SRV host (`modules/atlas-privatelink-dns`).
- **Per-project tags** (`Project = var.project_name`).
- The Atlas-PL SG ingress is **CIDR-based** on `var.vpc_cidr` (not SG-referenced)
  — every workload inside the shared VPC reaches Atlas via the VPCE without
  needing the Atlas-PL SG to know about per-project SGs.

The previous "two envs, not conditional flags" rationale still holds: each
`main.tf` is a self-contained description of what gets deployed; no `count`
indirection, no conditional outputs.

## Shared network is consumed via SSM Parameter Store

**Decision:** `envs/network` publishes the VPC + subnet + Atlas-PL identifiers
to SSM Parameter Store under prefix `/${SHARED_VPC_NAME}/${AWS_REGION}/`.
`envs/ec2` reads them via `data "aws_ssm_parameter"`. There is **no
`terraform_remote_state` coupling** between the two configs.

**Published params (StringList for the subnet IDs, String otherwise):**

```
/${SHARED_VPC_NAME}/${region}/vpc_id
/${SHARED_VPC_NAME}/${region}/vpc_cidr
/${SHARED_VPC_NAME}/${region}/public_subnet_ids
/${SHARED_VPC_NAME}/${region}/private_subnet_ids
/${SHARED_VPC_NAME}/${region}/atlas_pl_vpce_id
/${SHARED_VPC_NAME}/${region}/atlas_pl_vpce_dns_name
/${SHARED_VPC_NAME}/${region}/atlas_pl_security_group_id
/${SHARED_VPC_NAME}/${region}/atlas_endpoint_service_name
/${SHARED_VPC_NAME}/${region}/atlas_private_link_id
```

**Why SSM and not `terraform_remote_state`?** `terraform_remote_state` requires
the consumer to have S3 read perms on the producer's state file (which contains
secrets — Atlas API keys, etc.). SSM is a tighter, value-only contract;
consumers only see the IDs they need.

**Why `${SHARED_VPC_NAME}` is a tfvar with no default?** The value is sourced
from `.env` (`export SHARED_VPC_NAME="shared-network"`) and threaded through
`deploy-network.sh` and `deploy.sh` into the generated `terraform.tfvars`. No
default in `variables.tf` means the value cannot silently come from a Terraform
fallback — same convention as `PROJECT_NAME`, `ENVIRONMENT`, `VPC_CIDR`.

**Phase 3b precheck in `deploy.sh`:** before generating `envs/ec2`'s
`terraform.tfvars`, we verify the SSM params exist and emit
"Run `./deploy/scripts/deploy-network.sh` first" if they don't. This gives
a friendlier error than terraform's `ParameterNotFound`.

## IAM roles are all Terraform-managed (new account)

**Decision:** Every IAM role the stack needs is created by Terraform —
including the Bedrock KB role
(`<project_name>-bedrock-kb-<environment>-role`), EC2 instance role,
Lambda execution role, AgentCore Gateway role.

**Why:** The current AWS account is a greenfield POC account (no SSO role
constraint). The credentials we apply with have full `iam:Create*` /
`iam:Delete*` access, so the pre-Terraform workaround of referencing existing
roles via `data "aws_iam_role"` is no longer needed.

**Historical note (no longer applies):** In the original account, the SSO
PowerUser role lacked `iam:CreateRole` / `iam:DeleteRole`. Three roles created
out of band by bash scripts — the Bedrock KB role (then literal
`bedrock-kb-ts-role`, now project+env-derived), `ts-bedrock-kb-creator`,
`ts-aoss-index-creator` — were referenced via data sources with a provider
alias (`aws.bedrock_kb_creator`) that assumed the helper role for the KB
`iam:PassRole` requirement. That machinery has been removed. If we ever move
back to an SSO-constrained account we'll need to re-introduce it — keep the
pattern in mind.

## Bedrock KB and AgentCore — native AWS provider resources (provider 6.27+)

**Decision (2026-05-14, Phase 8):** All four modules
(`bedrock-kb`, `agentcore-memory`, `agentcore-gateway`,
`agentcore-agent-runtime`) now use native `aws_*` resources from the AWS
provider 6.x (>= 6.27). The previous `null_resource` + AWS CLI shims and
their JSON state files (`.kb-state.json`, `.memory-state.json`,
`.gateway-state.json`, `.runtime-state-*.json`) are gone.

**Resources used:**

- `aws_bedrockagent_knowledge_base` + `aws_bedrockagent_data_source`
  (provider supports `MONGO_DB_ATLAS` storage as of v6.x)
- `aws_bedrockagentcore_memory` (provider v6.18.0+)
- `aws_bedrockagentcore_gateway` + `aws_bedrockagentcore_gateway_target`
  (provider v6.17.0+)
- `aws_bedrockagentcore_agent_runtime` (provider v6.17.0+); the DEFAULT
  endpoint is auto-created by AWS, so we don't manage it explicitly.

**Why now:** AWS shipped native support for all four resources in provider
v6.17–v6.18 (October 2025). Migrating removed ~600 lines of bash that
duplicated Terraform's lifecycle management.

**Remaining `null_resource`:**

- `bedrock-kb/main.tf:null_resource.ingestion` — `start-ingestion-job` is
  an *action*, not a resource (no `aws_bedrockagent_ingestion_job` exists).
  Triggered on first apply and whenever S3 doc etags change.
- `bedrock-kb/main.tf:null_resource.ensure_collection` — wraps the bun
  `ensure-collection.ts` script that creates the Atlas collection before
  the vector index is built. There is no native Terraform resource for
  "create a MongoDB collection."

These two are intentionally minimal and don't carry their own state
files — Terraform infers reruns purely from the resource trigger inputs.

**Destroy:** `terraform destroy` now tears the KB / Memory / Gateway /
Runtime resources down through their native delete operations. No more
`destroy-*.sh` companion scripts.

### AgentCore runtime artifact mode

**Decision:** `modules/agentcore-agent-runtime` supports both artifact types:

- `deployment_mode = "code"` (default): S3 direct-code artifact (`codeConfiguration`)
- `deployment_mode = "container"`: ECR image (`containerConfiguration`)

`deploy.sh` builds `api/src/agent-runtime-code.ts` into `agent-runtime-code.js`,
zips it, uploads it to the shared S3 bucket, and passes that key to Terraform
through `agentcore_code_artifact_prefix`.

**Why:** Keeps TypeScript development ergonomics while aligning runtime artifact
management with S3-first deploy strategy. Container mode remains available for
cases requiring custom base images or larger artifacts.

## Lambda MCP packaging

**Decision:** The Lambda code lives in `lambda/mongodb-mcp/` at the repo root.
Terraform zips it with `archive_file`, uploads the zip to the shared S3 bucket
(`aws_s3_object`), and deploys the Lambda from `s3_bucket` + `s3_key`.
`bun install` is run **outside** Terraform (Phase 5 of `deploy.sh`) so
`node_modules/` is present on disk before the archive is built.

This gives us an artifact-driven deploy path and reproducibility from object
storage, aligned with the S3-first artifact strategy.

**Why outside Terraform?** Running `bun install` from a `null_resource` would
couple the Lambda's dependency graph to every `terraform plan`, slow down plans,
and add a failure mode that has nothing to do with infrastructure. The archive
hash is computed from the directory contents, so re-installing `node_modules`
always produces a new zip when dependencies change.

**Why a custom handler instead of the `mongodb-mcp-server` npm package?**
AgentCore Gateway invokes Lambda targets via the Lambda API, not over HTTP.
Gateway sends a Lambda invocation event containing the tool name + parameters.
A stock `mongodb-mcp-server` expects an HTTP MCP protocol on stdin/stdout.
`index.mjs` is ~150 lines that bridges the two: parses the Gateway event,
dispatches to `mongodb_query` / `mongodb_vector_search` / `mongodb_aggregate`,
returns the response in the shape Gateway expects.

## EC2 shell access: SSM only, no SSH

**Decision:** The EC2 security group opens :3000 (API) and :8501 (UI) only.
Port 22 is intentionally omitted. `AmazonSSMManagedInstanceCore` is attached to
the EC2 instance role so `aws ssm start-session --target i-…` works out of the
box.

**Why:**
- No key-pair management (lost/leaked keys, `chmod` footguns, onboarding teammates).
- No exposed SSH on the public internet.
- SSM works through corporate proxies (outbound HTTPS only).
- All sessions are auditable via CloudTrail.

The `ec2_key_pair_name` variable still exists — pass one in if you need SSH for
a specific reason. The SG will need a matching ingress rule added back.

`deploy.sh` uses `aws ssm send-command` (not `scp` or `ssh`) to push `.env.live`
and to trigger `systemctl restart`. This keeps the "no SSH" promise end-to-end.

## Atlas PrivateLink lives in envs/network; per-cluster DNS lives in envs/ec2

**Decision:** `modules/atlas-privatelink` (AWS Interface VPCE + Atlas project
binding + Atlas-PL SG) is called from `envs/network` only. The per-cluster
Route 53 private zone + wildcard CNAME (which used to live in the same
module) moved to `modules/atlas-privatelink-dns`, called once per project from
`envs/ec2`. Local mode connects to Atlas over its public SRV hostname.

**Why this split?** The AWS Interface VPCE is per-VPC; Atlas itself only
allows one endpoint service per (Atlas project, region). Both naturally live
once-per-region. The Route 53 zone, however, is named after the cluster's
SRV host (e.g., `mycluster.dcysxk.mongodb.net`) — it's per-cluster, hence
per-project. The wildcard CNAME inside the zone points at the
**shared** VPCE's regional DNS name (read from SSM in `envs/ec2`).

**SG ingress is CIDR-based on `var.vpc_cidr`** — not SG-referenced — because
the Atlas-PL SG must not depend on the existence of any specific app SG. Any
workload inside the shared VPC reaches the Atlas VPCE without further IAM
or SG plumbing. The Atlas-PL SG only sits on the VPCE ENIs themselves, so
broad CIDR ingress is acceptable (it does not protect user-facing workloads).

**Why local mode skips PrivateLink:** PrivateLink only makes sense when a VPC
exists. For local development, the app runs on the laptop and reaches Atlas
from the laptop's public IP — the IP allowlist is added manually in the Atlas
console (Atlas org policy `ATLAS_RESOURCE_POLICIES_VIOLATION_NOT_AUTHORIZED`
blocks API-based allowlist edits on this account, so we can't manage it via
the provider).

## No OpenSearch Serverless

**Decision:** The Bedrock KB uses MongoDB Atlas (`MONGO_DB_ATLAS` storage type)
as its vector store.

**Why:** AOSS would add a second vector store to manage and pay for. Atlas is
already the primary data store and supports `$vectorSearch` natively. Unifying
the two keeps the data layer simple.

**Legacy:** The original standalone troubleshooting bootstrap created
an AOSS collection `troubleshooting-kb` (id `onfbkma236kj8lb1txi7`). That was
torn down when we moved to Atlas-as-vector-store. The stale
`ts-aoss-index-creator` IAM role survives (can't delete — see above).

## One shared S3 bucket, many prefixes

**Decision:** A single bucket — `${project_name}-${environment}-${account_id}` —
holds both Terraform state and KB source docs. Namespaced by prefix:

```
s3://<project>-<env>-<account_id>/
├── <env>/terraform.tfstate           # envs/local state
├── <env>/ec2/terraform.tfstate       # envs/ec2 state
└── kb-docs/docs/*.txt                # Bedrock KB source documents
```

The bucket is provisioned by the standalone `bootstrap/` configuration and
referenced by every other env via `shared_bucket_name` variables.

**Design direction update:** this bucket is also the preferred place for
deployable artifact metadata and non-container package assets (for example,
Lambda zip bundles and build manifests).

**Why one bucket?**
- Account IDs in the name prevent cross-account collisions.
- Versioning + SSE apply to everything uniformly.
- One bucket to forget about / clean up, not N.
- Separate state keys (`<env>/…`) still give per-environment isolation.

**Why `force_destroy = false` on the shared bucket?** It holds Terraform state.
`terraform destroy` on the envs never needs to touch the bucket itself. If you
actually want to delete it, run `destroy.sh --with-bootstrap` which will empty
it first. By default KB docs are just a prefix in this shared bucket.

**Optional dedicated KB-docs bucket (`KB_DOCS_BUCKET`).** Setting the
`KB_DOCS_BUCKET` env var (→ `kb_docs_bucket_name` tfvar → `module.bedrock_kb`)
makes the `bedrock-kb` module create and use a separate bucket for KB source
docs (still under the `kb-docs/docs/` prefix). That bucket uses
`force_destroy = true` because it holds no Terraform state, so it is cleanly
removed by a normal `terraform destroy`. Terraform remote state always stays on
the canonical shared bucket — this override never moves state.

## Secrets Manager recovery window = 0

**Decision:** `recovery_window_in_days = 0` on all `aws_secretsmanager_secret`
resources managed by Terraform.

**Why:** A non-zero recovery window blocks recreating a secret with the same
name for 7–30 days after deletion (AWS's scheduled-deletion hold). Zero means
force-delete, so `terraform destroy` followed by `terraform apply` the same day
works without hitting naming conflicts.

---

## KB source documents are version-controlled

**Decision:** `kb-docs/*.txt` lives in git. Each file is uploaded to S3 via
`aws_s3_object` + `for_each = fileset(...)`. The ingestion job re-runs when any
file's `etag` changes.

**Why:** Previously docs were inlined as heredocs in a standalone troubleshooting
bootstrap. Moving them to tracked files surfaces content
changes in git diffs and auto-syncs the KB on `terraform apply`.

## What is NOT in Terraform

| Component | Where it lives | Why |
|---|---|---|
| MongoDB document + vector-index seeding | `db-seeding/*.ts` (Bun scripts) | Data migration, not infra. Run after cluster is ready. |
| Atlas IP allowlist (local mode) | Atlas console, manual | Atlas org policy blocks API-based allowlist edits on this account. |
| Docker image build | `docker-build-push.sh` | Code → image is a CI concern; Terraform handles infra. AgentCore Runtime currently requires container URI (ECR). |
| `session-manager-plugin` install | Developer laptop, one-time | Not a cloud resource. |
| Bun install for Lambda | `deploy.sh` Phase 5 | See "Lambda MCP packaging" above. |
| `.env.live` generation | `deploy.sh` / `deploy-local.sh` | Reads Terraform outputs + secrets; lives outside state. |

Anything else the reference architecture requires **should** be Terraform. If you find yourself
editing a resource by hand in the console, file an issue — it's a signal that a
module is missing.

## Apply order cheat sheet

First-time setup in a new account:

```bash
# 1. Bootstrap shared S3 bucket (one-time per account+region; deploy-network.sh
#    and deploy.sh do this automatically if absent).
source .env
cd deploy/terraform/bootstrap
terraform init && terraform apply -var="account_id=$(aws sts get-caller-identity --query Account --output text)"

# 2. Apply the shared network (one-time per region per SHARED_VPC_NAME)
./deploy/scripts/deploy-network.sh --auto-approve

# 3. Apply a per-project env (reads shared VPC + Atlas-PL via SSM)
./deploy/scripts/deploy-local.sh             # envs/local
# or
./deploy/deploy-full-with-privatelink.sh --auto-approve    # envs/ec2
```

Day-to-day:

```bash
./deploy/scripts/deploy-network.sh --auto-approve   # envs/network (rare)
./deploy/scripts/deploy-local.sh                    # envs/local
./deploy/deploy-full-with-privatelink.sh --auto-approve           # envs/ec2

./deploy/scripts/destroy.sh --mode local            # tears down envs/local only
./deploy/scripts/destroy.sh --mode ec2              # tears down envs/ec2 only
./deploy/scripts/destroy.sh --mode network          # tears down envs/network — DANGER
```

The three envs share state bucket but have separate keys
(see "Three root configs" above), so each `destroy --mode X` is independent.
**Caveat:** do not destroy the network env while any per-project ec2 env in
the same region is still applied — those envs read SSM values published by
network and will fail the next plan.

## Migrating from the per-project VPC layout (one-time)

For deployments that pre-date the `envs/network` split, the existing
`envs/ec2` state still owns the VPC + Atlas-PL VPCE + Route 53 zone. Two
migration paths are available depending on how much downtime is acceptable.

---

### Option A — State-move (no EC2 downtime, recommended)

This path transfers VPC ownership from `envs/ec2` to `envs/network` without
destroying any cloud resources. The EC2 instance and Lambda stay live throughout.

**Why it works without downtime:** `terraform state rm` removes the resources
from the ec2 state file without deleting them from AWS. `envs/network` then
creates a fresh set of shared resources (a new VPC / new VPCE are created — see
the two-VPC note below). The ec2 apply that follows destroys the old orphaned VPC
and recreates EC2 + Lambda in the new shared one. For a POC this causes a brief
EC2 replacement, but the Atlas cluster, AgentCore, Bedrock KB, and Cognito are
untouched.

```bash
# 1. Apply the shared network — creates a NEW shared VPC + VPCE + SSM params.
#    (You temporarily have two VPCs. Atlas reuses its existing endpoint service
#    for the new VPCE; the old per-project VPCE becomes an orphan at this step.)
./deploy/scripts/deploy-network.sh --auto-approve

# 2. Remove the networking + Atlas-PL resources from the ec2 state WITHOUT
#    destroying them. This tells Terraform "I no longer own these" so the
#    subsequent apply does not try to delete them.
source .env
cd deploy/terraform/envs/ec2
terraform init -input=false -reconfigure \
  -backend-config="backend.hcl"

# Remove VPC + subnets + IGW + RT
terraform state rm module.networking.aws_vpc.main
terraform state rm 'module.networking.aws_subnet.public[0]'
terraform state rm 'module.networking.aws_subnet.public[1]'
terraform state rm 'module.networking.aws_subnet.private[0]'
terraform state rm 'module.networking.aws_subnet.private[1]'
terraform state rm module.networking.aws_internet_gateway.main
terraform state rm module.networking.aws_route_table.public
terraform state rm 'module.networking.aws_route_table_association.public[0]'
terraform state rm 'module.networking.aws_route_table_association.public[1]'

# Remove Atlas PrivateLink resources
terraform state rm module.atlas_privatelink.aws_vpc_endpoint.atlas
terraform state rm module.atlas_privatelink.mongodbatlas_privatelink_endpoint_service.atlas
terraform state rm module.atlas_privatelink.aws_security_group.atlas_pl
terraform state rm module.atlas_privatelink.null_resource.atlas_pl_lookup
terraform state rm module.atlas_privatelink.aws_route53_zone.atlas
terraform state rm module.atlas_privatelink.aws_route53_record.atlas_wildcard

cd "$OLDPWD"

# 3. Re-apply ec2 with the refactored config (reads SSM from envs/network).
#    Terraform will:
#      - Destroy the old orphaned per-project VPCE + VPC (now unowned by any
#        state after step 2 — they exist in AWS but not in any state file, so
#        you need to clean them up manually or let deploy.sh's -refresh-only
#        catch them). See note below.
#      - Create module.atlas_privatelink_dns (Route53 zone + CNAME).
#      - Replace EC2 instance + Lambda MCP in the new shared subnets.
./deploy/deploy-full-with-privatelink.sh --auto-approve
```

**Orphan cleanup after step 3:** after the state-rm the old VPC and per-project
Interface VPCE exist in AWS but are no longer tracked in any Terraform state.
They will not be destroyed automatically. Clean them up manually:

```bash
# Find and delete the old per-project Interface VPCE (tag: Project=<project_name>)
aws ec2 describe-vpc-endpoints \
  --region "$AWS_REGION" \
  --filters "Name=tag:Project,Values=${PROJECT_NAME}" \
  --query "VpcEndpoints[].VpcEndpointId" --output text \
  | xargs -I{} aws ec2 delete-vpc-endpoints --region "$AWS_REGION" --vpc-endpoint-ids {}

# Wait for the VPCE to reach deleted state, then delete the VPC (AWS will
# refuse if any ENIs or subnets still exist — the VPCE deletion clears them).
OLD_VPC_ID=$(aws ec2 describe-vpcs \
  --region "$AWS_REGION" \
  --filters "Name=tag:Project,Values=${PROJECT_NAME}" \
             "Name=tag:Name,Values=${PROJECT_NAME}-vpc-*" \
  --query "Vpcs[?VpcId!='$(aws ssm get-parameter \
    --region $AWS_REGION \
    --name "/${SHARED_VPC_NAME}/${AWS_REGION}/vpc_id" \
    --query Parameter.Value --output text)'].VpcId" \
  --output text)
# Review $OLD_VPC_ID carefully before deleting
aws ec2 delete-vpc --region "$AWS_REGION" --vpc-id "$OLD_VPC_ID"
```

### Option B — Destroy and redeploy (simpler, causes full downtime)

Use this when clean-slate is acceptable (POC / dev environment):

```bash
# 1. Tear down everything in envs/ec2 (EC2 + Lambda + Atlas cluster + AgentCore
#    + VPC + Atlas-PL VPCE — ~20 min)
./deploy/scripts/destroy.sh --mode ec2 --auto-approve

# 2. Apply the shared network (creates VPC + Atlas-PL + SSM params)
./deploy/scripts/deploy-network.sh --auto-approve

# 3. Re-apply ec2 (consumes shared VPC via SSM — fresh Atlas cluster is
#    provisioned; re-seed MongoDB and re-upload AgentCore artifacts)
./deploy/deploy-full-with-privatelink.sh --auto-approve
```

### Atlas endpoint service is always preserved

Both paths leave the Atlas endpoint service
(`com.amazonaws.vpce.<region>.vpce-svc-...`) intact. The
`modules/atlas-privatelink/scripts/discover-or-create-pl.sh` script reuses
the existing service on every apply (Atlas returns HTTP 409 on a duplicate
create and the script handles it). Atlas allows only one endpoint service per
(Atlas project, region), so this is required for idempotency.
